# Git-like DAG + LLM-synthesized Distillation/Narration + Judge-verified Fidelity for Agent Memory

## Abstract

Agent memory systems need more than vector recall: they need mutation history, derived representations, rollback semantics, and interop boundaries that survive migration. MNEMOS models each memory as a Postgres-backed record with a content-addressed version DAG, where `memory_versions` and `memory_branches` record every mutation and every derived artifact with explicit parentage. Compression and narration are represented as branch-level derivations, with APOLLO and ARTEMIS producing contest candidates and a fidelity judge enforcing a configurable quality floor before a variant can become the served representation. MORPHEUS extends the same memory substrate with a divergent pipeline for replay, clustering, consolidation, synthesis, and extraction, while PERSEPHONE moves cold unrecalled memories to compressed archival storage with live stub pointers. Federation, archival, unified LLM routing, and MCP model discovery close the feedback loop by making memory state portable, operationally inspectable, and available to downstream agents without treating any one storage format as the final ecosystem standard.

## 1. Introduction

Long-running agents need durable context across sessions, tools, models, and
operators. The common implementation is a vector store plus a retrieval
routine: embed text, search for nearest neighbors, and inject a short result set
into the next model call. That design solves the first recall problem, but it
does not solve memory evolution. Once an agent can update, compress, narrate,
merge, archive, or exchange memories with peer systems, the memory layer needs
lineage and verification. A retrieved sentence is not enough; the system also
needs to know where it came from, which version superseded it, which derived
forms are faithful, which rows were generated by background synthesis, and which
formats can carry the state across system boundaries.

Existing systems cover important portions of this space. MemGPT introduced an
OS-inspired memory hierarchy for managing context beyond the active prompt.
Letta builds on that lineage with persistent agents and explicit memory
management. Mem0 focuses on a production memory layer that extracts, updates,
and retrieves user and agent facts through a developer-facing API. MemPalace
emphasizes local-first verbatim storage, semantic retrieval, knowledge graph
operations, and a memorable MCP vocabulary. Cognee frames memory as a data and
knowledge engine, building graph structures over documents and interactions.
Graphiti provides a temporal knowledge graph that can be updated incrementally
as relationships change.

The gap MNEMOS fills is not "another recall engine." The design target is a
memory substrate whose primary object is a versioned, derivable, auditable
memory record. The substrate should support raw memories, compressed variants,
prose narration, generated summaries, extracted KG triples, archival stubs, and
federated imports without collapsing them into one mutable row. It should also
make loss boundaries explicit: compression must prove fidelity, federation must
preflight schema compatibility, and interchange formats must preserve sidecars
or say exactly what is dropped.

This paper describes the MNEMOS v5.0.0 design as implemented in the local
codebase. It is a design paper, not a completed evaluation report. The
evaluation work described later should be run by the PROTEUS barrage against
fixed corpora and fixed deployment profiles.

## 2. Architecture Overview

MNEMOS uses Python services over Postgres, with pgvector used for embedding
search in the production profile. The row model begins with a `memories` table
for current live state and extends into version, compression, federation, and
archival tables. The central design choice is that memory mutation is recorded
as a DAG rather than as a linear append-only changelog.

The versioning substrate is defined in
[`db/migrations_v2_versioning.sql`](../../db/migrations_v2_versioning.sql).
The migration creates `memory_versions` and a snapshot trigger that records
create, update, and delete events. Each version row carries `memory_id`,
`version_num`, `content`, ownership and namespace fields, `change_type`, a
`parent_version_id`, a `branch`, and a `commit_hash`. `memory_branches` tracks
the current head for each named branch. The parent pointer is the local lineage
edge; the branch head is the mutable reference; the version row is the immutable
event record.

That schema gives a concrete testable claim: the `memory_versions` DAG records
every ordinary mutation with a `parent_version_id` chain back toward the create
commit on the branch being written, while `memory_branches` records which node
is currently served for that branch. A derived subsystem does not need a
parallel audit table to explain its output. It needs to write a version row with
the right parent and move a named branch head inside the same transaction.

Commit hashes are deterministic from version content and derivation context
where the explicit derivation path owns the write. For compression derivations,
[`mnemos/domain/compression/contest_store.py`](../../mnemos/domain/compression/contest_store.py)
computes the compression commit hash from the parent commit hash, variant
content, and branch name. That gives derived artifacts a content-addressed
identity tied to the source version, so re-persisting the same artifact on the
same branch is idempotent rather than a new unrelated mutation.

This is Git-like in the specific sense that the design separates immutable
version objects from named branch heads. It is not a claim that every Git
operation exists. Three-way merge conflict resolution, federated quorum writes,
and semantic versioning of derived artifacts under model upgrades remain open
problems.

## 3. The DAG Branch Model

The canonical branch is `main`. It stores the ordinary mutation history of a
memory: create, update, and delete snapshots. Reads of the current memory row
are the operational hot path, while `memory_versions` preserves the historical
chain.

Compression derivations add two conventional branches:

| Branch | Role | Typical producer |
|---|---|---|
| `main` | Source memory history | user, agent, import, federation |
| `distilled` | Dense or structured compression artifact | APOLLO |
| `narrated` | Prose-shaped narration or extractive variant | ARTEMIS or narration path |

[`db/migrations_v4_2_compression_dag.sql`](../../db/migrations_v4_2_compression_dag.sql)
formalizes this by expanding `memory_versions.change_type` to include
`compress` and documenting `distilled` and `narrated` as branch conventions. The
branch names are not heavy schema constraints. They are operational contracts:
callers that ask for dense APOLLO output should look at the distilled branch;
callers that ask for prose should look at the narrated branch.

The compression persistence path does not mutate `memories.content`. It writes
the contest audit rows, upserts the current winning variant, and inserts a child
version on the appropriate derived branch. The parent is the source `main`
branch head. The derived branch head then moves to the new compression version.
This keeps raw source text available while allowing derived forms to be served
on hot paths through content negotiation or MCP parameters.

The design generalizes beyond compression. A future branch can represent an
operator-reviewed synthesis, a model-upgraded dense encoding, or a generated
explanation, provided it records parentage and preserves the source branch.

## 4. Distillation and Judge-verified Fidelity

Compression is an operator-batched contest rather than an automatic write-time
side effect. The operator enqueues memories, workers drain
`memory_compression_queue`, engines emit candidates, and
`memory_compression_candidates` records every attempt. The current winning
variant is stored in `memory_compressed_variants`. The operator flow is
summarized in [`docs/COMPRESSION.md`](../COMPRESSION.md), and the schema is
defined in [`db/migrations_v3_1_compression.sql`](../../db/migrations_v3_1_compression.sql).

APOLLO and ARTEMIS are the built-in engines under
[`mnemos/domain/compression/`](../../mnemos/domain/compression). APOLLO is the
dense, schema-aware path; ARTEMIS is a CPU-only extractive engine that favors
identifier preservation and prose-shaped output. Third-party engines plug into
the `CompressionEngine` ABC in
[`mnemos/domain/compression/base.py`](../../mnemos/domain/compression/base.py).

The contest runner in
[`mnemos/domain/compression/contest.py`](../../mnemos/domain/compression/contest.py)
applies a scoring profile with three major controls: quality score,
compression-ratio reward, and speed factor. The critical safety mechanism is
`quality_floor`. Built-in profiles set different floors, for example 0.70 for
`balanced`, 0.80 for `quality_first`, and 0.60 for `speed_first`. Candidates
below the configured floor are rejected with `reject_reason='quality_floor'`
before composite scoring. A candidate therefore cannot win solely because it is
small or fast if it fails the fidelity threshold.

ARTEMIS can self-report a calibrated quality score for extractive outputs.
APOLLO and other dense forms can be scored by the judge path in
[`mnemos/domain/compression/judge.py`](../../mnemos/domain/compression/judge.py).
When a judge is supplied, the contest runner narrates APOLLO dense output back
to prose, asks the judge to rate fidelity against the original, replaces the
engine score with the judge score, and records the judge model. The engine's
own score is preserved in the manifest for audit. Judge failure does not fail
the contest closed; the runner falls back to the engine score and records the
candidate's normal audit trail.

This separation matters because compression is not merely storage reduction.
For agent memory, a compressed artifact may be injected into future reasoning
contexts. A lower-token representation that drops a constraint, identifier,
negation, or date can create a behavioral regression. The quality floor makes
that trade explicit and configurable.

## 5. MORPHEUS Divergent Dream-state Pipeline

Compression is convergent: it attempts to preserve one memory in fewer or more
structured tokens. MORPHEUS is divergent: it processes groups of memories into
clusters, summaries, consolidation markers, and extracted KG triples. The run
orchestrator lives in
[`mnemos/domain/morpheus/runner.py`](../../mnemos/domain/morpheus/runner.py).

The implemented pipeline is:

`REPLAY -> CLUSTER -> CONSOLIDATE -> SYNTHESISE -> EXTRACT -> COMMIT`

REPLAY selects the run window. CLUSTER groups similar memories, with cluster
payloads stored on the `morpheus_runs` row for later phases. CONSOLIDATE can
soft-link near duplicates to a canonical memory by setting `consolidated_into`
and making originals read-only through permission-mode changes. SYNTHESISE
creates generated summary memories with `provenance='morpheus_local'` and
`source_memories` pointing back to inputs. EXTRACT mines latent KG triples from
eligible prose memories and tags those triples with the run id.

The rollback contract is the important architectural feature. The foundation
migration
[`db/migrations_v3_3_morpheus.sql`](../../db/migrations_v3_3_morpheus.sql)
adds `morpheus_runs`, `memories.morpheus_run_id`, `source_memories`, and
`provenance`. Later migrations add consolidation and extraction columns:
[`db/migrations_v4_2_morpheus_consolidate.sql`](../../db/migrations_v4_2_morpheus_consolidate.sql)
and
[`db/migrations_v4_2_morpheus_extract.sql`](../../db/migrations_v4_2_morpheus_extract.sql).
Each row created or mutated by a run is tagged with `morpheus_run_id`; triples
created by extraction use `extracted_by_run_id`. Rollback deletes run-created
summary rows, removes extracted triples from that run, and restores
consolidated originals from metadata. The design avoids a long database
transaction across slow LLM phases while preserving scoped reversibility.

## 6. Federation and Portability

MNEMOS federation is pull-based, with any NATS fast path treated as later
work. The schema-compatibility preflight is exposed at
`GET /v1/federation/schema` in
[`mnemos/api/routes/federation.py`](../../mnemos/api/routes/federation.py).
The response includes `mnemos_version`, `schema_signature`, and
`migrations_fingerprint`, plus informational core fields. A peer can query this
before syncing and decide whether strict or permissive compatibility applies.

CHARON is the portability subsystem. There is no `docs/CHARON.md` in this
tree; the current documentation is split across
[`docs/MEMORY_EXPORT_FORMAT.md`](../MEMORY_EXPORT_FORMAT.md),
[`mnemos/tools/README.md`](../../mnemos/tools/README.md), and the adapter
modules under [`mnemos/tools/adapters/`](../../mnemos/tools/adapters). MPF
v0.1.1 carries memory records plus sidecars such as `kg_triples`,
`memory_versions`, and `compression_manifest`. The adapter set covers Mem0,
Letta, Graphiti, Cognee, and MemPalace as import/export peers, with each adapter
mapping source-specific identity, content, graph, or event models into MPF.

The posture is alignment, not format competition. MPF is frozen as the
MNEMOS-side working shape while MIF matures as a broader memory interchange
format. The design lessons from MPF are directly relevant to MIF: sidecars need
record-stable ids; derived variants need lineage; version history preservation
needs authority checks; schema compatibility needs a preflight rather than a
best-effort import that fails halfway through a corpus.

## 7. PERSEPHONE Archival

PERSEPHONE moves cold memories out of the live working set while preserving a
visible pointer. The migration
[`db/migrations_v4_2_persephone.sql`](../../db/migrations_v4_2_persephone.sql)
creates `memory_archive` with zstd-compressed payload bytes and adds
`memories.archived_at`. Archived memories keep a live row with an archive
marker, so reads, federation feeds, and operator tools can see that the memory
exists without carrying the full payload in the hot table.

Eligibility is driven by recall-tracking columns added in
[`db/migrations_v3_3_recall_tracking.sql`](../../db/migrations_v3_3_recall_tracking.sql):
`recall_count` and `last_recalled_at`. The admin routes in
[`mnemos/api/routes/admin.py`](../../mnemos/api/routes/admin.py) expose root
sweeps, explicit archive, explicit restore, and status. The status endpoint
reports archived count and cold-set age signals. Restore is explicit and gated
by root or owner authority; archival does not auto-restore on ordinary reads in
this slice.

This design treats archival as memory state evolution rather than deletion. The
stub pointer preserves identity and federation visibility, while the compressed
archive table reduces the active storage and indexing footprint for memories
that are not being recalled.

## 8. PANTHEON, IRIS, and the Feedback Loop

PANTHEON is MNEMOS's unified LLM facade, documented in
[`docs/PANTHEON.md`](../PANTHEON.md) and implemented under
[`mnemos/domain/pantheon/`](../../mnemos/domain/pantheon). The v0.2 slice adds
per-session caps for `consultation_only` models, best-effort
`pantheon_routing` memory writes, rolling-window adaptive routing for `auto:*`
aliases, and route explanation output. The memory writes make model routing
observable to MNEMOS rather than leaving it as transient gateway state.

IRIS is the discovery role over the model catalog. In the current v5.0.1 codebase,
the discovery role is surfaced through unified MCP model tools rather than a
separate IRIS server. [`mnemos/mcp/tools/models.py`](../../mnemos/mcp/tools/models.py)
exposes `pantheon_list_models` and `pantheon_route_explain`, giving MCP callers
structured access to the PANTHEON catalog and routing decisions.

This is intentionally brief in this paper because the memory feedback loop is
the relevant point: routing decisions become memories, model availability is
discoverable by agents, and future memory work can use that operational record
when selecting compression judges, synthesis models, or extraction models.

## 9. Evaluation Design

This paper does not present final benchmark data. The correct evaluation should
come from the PROTEUS barrage using fixed corpora, fixed model settings, and
repeatable deployment profiles. Three evaluation tracks are most important.

First, recall fidelity at different compression ratios. A corpus should be
stored raw, distilled through APOLLO, narrated through ARTEMIS, and then queried
under identical retrieval workloads. Metrics should include exact answer
fidelity, identifier preservation, contradiction rate, and query-level recall at
K. The independent variable is not only compression ratio; it is also branch
choice (`main`, `distilled`, `narrated`) and quality floor.

Second, latency and load. Compression and archival can reduce bytes and tokens,
but the branch and variant lookups introduce joins and content negotiation.
Evaluation should measure p50, p95, and p99 latency for `GET /v1/memories/{id}`,
search, federation feed, and MCP `get_memory` under mixed raw/compressed
requests. PERSEPHONE sweeps and MORPHEUS runs should be included as background
load to verify that operator-batched work does not destabilize hot reads.

Third, federation round-trip lossless rate. A corpus containing memories,
version histories, KG triples, compression manifests, archived markers, and
foreign adapter imports should be exported through MPF, imported into a peer,
re-exported, and compared field by field. The lossless rate should be reported
per surface, not as a single aggregate: memory payloads, version rows, KG
triples, compression manifests, archive markers, and source-specific metadata
have different failure modes.

Future evaluation work should add adversarial cases: conflicting updates on
federated peers, model upgrades that change dense encoding semantics, very old
archival stubs, and cross-cluster KG triple duplication after MORPHEUS EXTRACT.

## 10. Related Work

MemGPT introduced virtual context management for LLMs, using an OS analogy to
page information between active context and longer-term storage. Its core
contribution is context control under fixed context windows. MNEMOS inherits
the need for hierarchical memory but places more emphasis on persistence
semantics: every mutation is recorded in a version DAG, and derived
representations are branch artifacts rather than hidden prompt state.

Letta, the successor lineage to MemGPT, focuses on stateful agents that can
learn, preserve memory, and move across environments. Its model is agent-first:
memory supports a persistent agent runtime. MNEMOS is substrate-first: agents
are important clients, but the memory store also has to serve federation,
MCP-compatible tools, archival, and operator audit.

Mem0 provides a production-oriented memory layer for AI applications, with APIs
for adding, extracting, updating, and retrieving memories across users and
agents. Its strength is developer integration and managed memory behavior.
MNEMOS fills a different gap by making lineage, compression contests, schema
preflight, and sidecar-preserving portability first-class system features.

Graphiti provides temporally aware knowledge graphs that can evolve as facts
and relationships change. MNEMOS uses KG triples as one sidecar and MORPHEUS
can extract triples from prose, but the primary unit remains the memory record
and its version DAG. The gap filled by MNEMOS is not richer temporal graph
reasoning; it is the coupling of graph sidecars to memory lineage,
compression, archival, and federation.

Cognee organizes documents and data into AI memory through graph construction,
embeddings, and search operations. It is closest to MNEMOS on the idea that
memory is more than a vector collection. MNEMOS differs by emphasizing
Postgres-backed mutation history, operator-batched derived artifacts, and an
interchange format that can carry version and compression sidecars.

MemPalace is a local-first memory system with verbatim storage, palace
terminology, semantic search, temporal KG operations, and MCP tools. It is a
strong fit for individual local workflows. MNEMOS targets the point where the
same vocabulary needs server-backed features: ownership, namespaces,
federation, HTTP APIs, version DAGs, archival, and audit. KNOSSOS exists to let
MemPalace-style tool calls use that backend without changing agent prompts.

KNOSSOS is not a competing memory model; it is a protocol adapter. It maps
MemPalace tool-name semantics onto MNEMOS routes and response shapes. Its
importance in this design is evidence that interop should happen at both
levels: file and envelope formats for bulk portability, and MCP vocabulary
adapters for live agent workflows.

## 11. Discussion and Future Work

Cross-cluster KG triple deduplication remains open. MORPHEUS EXTRACT can mine
triples from many memories, and federation can import triples from peers. The
system needs a principled deduplication pass that recognizes semantically
equivalent triples across clusters without deleting provenance. A likely design
is a canonical triple node with per-source assertion rows, but that introduces
new merge and invalidation semantics.

Semantic versioning of compressed variants under model upgrades is also open.
The current branch and commit-hash model records content identity and parentage,
but it does not define when an APOLLO output produced by a newer model should
supersede an older output if both pass the same quality floor. Future work
should treat compression-engine version, judge model, prompt version, and
schema version as an explicit compatibility tuple.

Federated quorum-write protocols are outside the current design. MNEMOS
federation is pull-based and eventually consistent. That is the correct default
for remote memory sharing, but collaborative multi-writer deployments may need
quorum writes, conflict detection, or operator-mediated merge queues. Adding
that without breaking the simple pull model will require a separate protocol
rather than a hidden behavior change in `/v1/federation/feed`.

Finally, MIF alignment should replace local format gravity with ecosystem
compatibility. MPF has useful lessons in sidecars, version preservation, and
compression manifests, but it should not become a permanent fork if MIF can
represent those concerns. The best outcome is a shared conformance suite that
lets MNEMOS, MemPalace, Mem0, Letta, Graphiti, Cognee, and future systems test
the same round-trip claims.
